# 검증 루프

`./scripts/claude/verify.sh`가 통과할 때까지 **고치고 다시 검증하는 것을 반복**한다.

추가 지시(있으면): $ARGUMENTS

## 현재 상태

!`./scripts/claude/verify.sh 2>&1 | tail -30`

## 루프 규칙

위 검증 결과에서 시작한다. **통과했다면 아무것도 고치지 말고 그 사실만 보고하고 끝낸다.**

실패했다면 다음을 반복한다:

1. **가장 먼저 실패한 것 하나**를 고친다. 여러 개를 한꺼번에 고치지 않는다 — 무엇이 무엇을 고쳤는지 알 수 없게 된다.
2. `./scripts/claude/verify.sh`를 다시 실행한다.
3. 통과하면 끝낸다. 실패하면 1번으로 돌아간다.

## 반드시 지킬 것

- **원인을 고친다. 검사를 고치지 않는다.** 검사 스크립트(`scripts/claude/`)를 수정해서 통과시키는 것은 부정행위다. 검사 자체가 틀렸다고 판단되면 루프를 멈추고 근거와 함께 사람에게 물어본다.
- **같은 실패가 3회 반복되면 멈춘다.** 접근이 통하지 않는다는 뜻이다. 무엇을 시도했고 왜 막혔는지 보고하고 사람의 판단을 구한다.
- **콘텐츠를 지어내지 않는다.** assets의 QA JSON은 사용자가 만든 학습 자료다. `answer_en` 같은 영어 답변이 비어 있으면 임의로 창작하지 말고 사용자에게 알린다. 오픽 학습의 정확성이 걸린 문제다.
- **범위를 넓히지 않는다.** 검증을 통과시키는 데 필요한 것만 고친다. 눈에 띈 다른 개선점은 고치지 말고 마지막에 따로 언급한다.
- 아키텍처 위반은 CLAUDE.md의 '코드 리뷰 체크리스트'가 근거다. Domain은 Data를 몰라야 하고, Presentation도 Data를 몰라야 한다. 우회하지 말고 올바른 방향(인터페이스는 domain에, 구현은 data에)으로 고친다.

## 끝낼 때

무엇이 실패했고 무엇을 고쳤는지 한국어로 간결하게 보고한다. 고치지 못한 것이 있으면 숨기지 말고 그대로 말한다.
